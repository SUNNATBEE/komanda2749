# GitHub — шпаргалка для команды (общая)

> Этот файл дублируется в инструкции каждого студента с **вашим именем ветки**.

## Основные понятия

| Термин | Что это |
|--------|---------|
| **Repository (репозиторий)** | Проект на GitHub — все файлы и история изменений |
| **Branch (ветка)** | Отдельная «линия» разработки. Каждый работает в своей ветке |
| **main / master** | Главная ветка — сюда попадает только проверенный код |
| **commit** | Сохранение ваших изменений с описанием |
| **push** | Отправка коммитов на GitHub |
| **pull** | Скачивание последних изменений с GitHub |
| **merge** | Объединение вашей ветки с `main` (через Pull Request) |
| **Pull Request (PR)** | Запрос на слияние: «проверьте мой код и добавьте в main» |

## 1. Первый раз — клонировать проект

```bash
git clone https://github.com/ВАШ-ОРГАНИЗАТОР/2749.git
cd 2749
npm install
npm run dev
```

## 2. Перед началом работы — обновить main

```bash
git checkout main
git pull origin main
```

**Зачем:** получить последние изменения команды, чтобы не работать на устаревшем коде.

## 3. Создать свою ветку

```bash
git checkout -b feature/ВАШЕ-ИМЯ
```

Пример: `git checkout -b feature/mansur`

**Зачем:** ваши правки изолированы — вы не ломаете код других.

## 4. Работать и сохранять (commit)

```bash
git status
git add .
git commit -m "feat: добавил hero-секцию на главной"
```

**Правила сообщений commit:**
- `feat:` — новая функция
- `fix:` — исправление бага
- `style:` — вёрстка, CSS, без логики

## 5. Отправить на GitHub (push)

```bash
git push -u origin feature/ВАШЕ-ИМЯ
```

Первый раз `-u` запоминает связь ветки. Дальше достаточно:

```bash
git push
```

## 6. Pull Request — запрос на merge

1. Откройте репозиторий на GitHub
2. Появится кнопка **Compare & pull request** — нажмите
3. Опишите, что сделали
4. Назначьте ревьюера (тимлид / преподаватель)
5. После одобрения нажмите **Merge pull request**

**Merge** = ваш код попадает в `main`.

## 7. После merge — обновить локальный main

```bash
git checkout main
git pull origin main
git branch -d feature/ВАШЕ-ИМЯ
```

## 8. Если кто-то уже изменил main — подтянуть изменения

Пока вы в своей ветке:

```bash
git checkout feature/ВАШЕ-ИМЯ
git fetch origin
git merge origin/main
```

Или (альтернатива):

```bash
git pull origin main
```

Если Git показывает **conflict** — откройте указанные файлы, исправьте конфликт вручную, затем:

```bash
git add .
git commit -m "fix: разрешил конфликт с main"
git push
```

## Чего НЕ делать

- Не пушить напрямую в `main` без согласования
- Не менять `Header.jsx`, `Footer.jsx`, `AppRouter.jsx` без разрешения
- Не делать `git push --force` на общие ветки
- Не коммитить `.env`, пароли, личные ключи

## Полезные команды

```bash
git branch              # список веток
git log --oneline -5    # последние 5 коммитов
git diff                # что изменилось (ещё не добавлено)
git stash               # временно спрятать изменения
git stash pop           # вернуть спрятанное
```
